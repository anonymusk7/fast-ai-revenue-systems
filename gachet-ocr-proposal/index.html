Architecture sketch

Keep extraction reviewable before it becomes automation

Tax documents are sensitive and inconsistent, so the first version should make every field traceable to the source document before data is exported into tax software.

1. Intake

n8n webhook or upload route receives PDF/image/CSV files and stores metadata, source hash, and processing state.

2. OCR

Google Vision or Textract adapter returns text blocks, confidence, page references, and raw output for audit.

3. Parse

Gemini extraction prompt maps known tax fields into JSON with confidence, missing-field flags, and source citations.

4. Review

HTML/JS preview shows document, extracted fields, copy buttons, edit states, and validation warnings.

5. Export

Postgres stores normalized records and exports Drake-ready CSV only after human review approves the row.

Risk controls

Designed for live-call reliability

Sensitive Documents

Use masked sample documents for the first milestone. If real tax documents are required, keep a written access path and avoid unnecessary retention.

Extraction Confidence

Do not auto-export low-confidence fields. Surface confidence, source page, and missing-field warnings in the review UI.

Operational Fallback

If OCR or LLM parsing fails during a call, the queue should preserve the source file and let staff enter the needed fields manually.
