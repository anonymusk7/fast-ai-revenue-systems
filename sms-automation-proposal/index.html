Scenario architecture

Separate message handling from expensive decisions

The system should keep inbound capture, media processing, search orchestration, response generation, and re-engagement state separate enough to test and recover each part.

1. Inbound Webhook

Receive Sinch SMS/MMS payloads, normalize phone numbers to E.164, and create a request record.

2. Media Handler

Download MMS images, validate file size/type, base64 encode, and attach media metadata for AI processing.

3. User Lock

Use a per-phone data-store lock with timeout cleanup to prevent overlapping searches from the same user.

4. Parallel APIs

Fire Rainforest and SerpApi branches in parallel, record response timing, and normalize the result schema.

5. Claude Reply

Send structured context to Claude Haiku and return a concise SMS-safe answer with fallback copy.

6. Tracking

Update users, analytics, savings, redirect tracking, and re-engagement state without blocking the reply.

Reliability controls

Failure paths should be designed before the demo

Concurrency

Every inbound message checks a user-level lock first. Stale locks should expire so a failed scenario does not trap a user forever.

Cost Boundaries

Skip expensive API or AI calls when payloads are invalid, duplicate, too large, or already in-flight.

Demo with synthetic SMS, MMS, slow API, failed API, duplicate message, and re-engagement cases before touching live traffic.
